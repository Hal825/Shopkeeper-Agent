# 在 1.4GB 内存上复现 NL2SQL 语义索引全链路

> 从 Docker 崩溃到 4/4 命中率，一次完整的工程踩坑与复盘。

---

## 背景与目标

我们的任务是复现一个 **NL2SQL 语义理解引擎的索引层**，需要实现：

- 从配置文件和真实数仓中提取表/字段/指标的元数据
- 存入 MySQL
- 用本地 Embedding 模型生成字段和指标的语义向量，写入 Qdrant
- 将字段的真实取值写入 Elasticsearch 做全文索引
- 最终支持 “销售额” → `order_amount` 这样的语义字段检索，以及 “华东” → `region_name = '华东'` 这样的值匹配

整个流程由一个 `MetaKnowledgeService` 串联，通过 `build_meta_knowledge.py` 驱动。

原始代码是完整而规范的，但我们的运行环境只有 **2.4GB 内存（实际可用约 1.4GB）**，同时跑 MySQL、Qdrant、Elasticsearch、Kibana 以及一个 Embedding 推理服务显然不可能。我们必须对环境进行深度适配和改造。

---

## 环境限制与改造策略

| 组件 | 原始方案 | 受限环境方案 |
|------|---------|------------|
| Embedding | TEI 容器 (HuggingFace Text Embeddings Inference) | 本地加载 `bge-small-zh` 模型，直接使用 `sentence-transformers` + `langchain_huggingface` |
| 向量数据库 | Qdrant 容器 (限制 128MB) | 保持不变，但调小 batch size 防止 OOM |
| MySQL | 两个实例 (Meta + DW) | 合并为单实例，限制内存 128MB |
| Elasticsearch | 官方镜像 | 构建带 IK 插件的镜像，Heap 限制 256MB，内存限制 512MB |
| Kibana | 开发调试用 | 需要时启动，平时关闭 |

我们放弃了 TEI 服务，改为在 Python 进程内加载 `BAAI/bge-small-zh` 模型。这避免了容器崩溃和网络调用开销，但也带来了异步调用适配问题——`HuggingFaceEmbeddings` 是同步接口，而我们整个构建链路都是异步的，需要用 `asyncio.to_thread` 或 `run_in_executor` 包装。

---

## 踩过的坑与修复实录

### 1. TEI 服务频繁 502：`chunk_size must not be zero`

- **现象**：第一个查询能成功，第二个请求立即导致 TEI 容器崩溃重启，日志中出现 `panicked at ... chunk_size must not be zero`。
- **原因**：`cpu-1.8` 镜像存在内部缺陷，在特定输入或并发请求下会触发零分块错误。
- **尝试的解决**：降低并发请求数、减小最大 token 数、降级到 `cpu-1.4` 镜像（但网络拉取失败）。
- **最终方案**：**彻底弃用 TEI**，改为在本地 Python 进程中加载 `sentence-transformers` 模型，从根本上消除服务不稳定的问题。

### 2. 虚拟环境缺失依赖的无底洞

- **现象**：`ModuleNotFoundError` 反复出现，包括 `sentence_transformers`、`loguru`、`asyncmy` 等。
- **原因**：重建的虚拟环境是干净的，没有安装任何第三方包。
- **解决**：在项目根目录找到 `requirements.txt`，执行 `pip install -r requirements.txt` 一次性补齐所有依赖。注意：必须在激活的虚拟环境中执行。

### 3. `EmbeddingClientManager` 类用错

- **现象**：`'HuggingFaceEndpointEmbeddings' object has no attribute 'aembed_query'`
- **原因**：原代码中使用 `HuggingFaceEndpointEmbeddings` 连接远程推理服务，而我们换成了本地模型，却只改了实例化，没有改类导入和异步方法支持。
- **解决**：
  - 导入改为 `from langchain_huggingface import HuggingFaceEmbeddings`
  - 在 `EmbeddingClientManager` 中添加 `aembed_query` 和 `aembed_documents` 异步方法，内部使用 `asyncio.get_running_loop().run_in_executor()` 包装同步调用。

### 4. MySQL 入库主键冲突

- **现象**：`IntegrityError: (1062, "Duplicate entry 'dim_region.region_id' for key 'column_info.PRIMARY'")`
- **原因**：多次运行 `build()` 时，Meta MySQL 中已存在旧数据，而插入逻辑是 `INSERT`，没有使用 `REPLACE` 或先删除。
- **解决**：在 `_save_tables_to_meta_db` 和 `_save_metrics_to_meta_db` 方法中，先执行 `DELETE FROM table_name` 清空旧数据，再写入新数据。注意必须使用 `session.execute(text(...))` 并手动 `commit()`。

### 5. 方法忘记返回导致 `NoneType` 错误

- **现象**：`TypeError: 'NoneType' object is not iterable`，发生在 `_save_column_info_to_qdrant` 中遍历 `column_infos` 时。
- **原因**：修改 `_save_tables_to_meta_db` 时，在末尾清空和写入逻辑后，**忘了写 `return column_infos`**，导致函数返回 `None`。
- **解决**：补上 `return column_infos`（以及指标方法中的 `return metric_infos`）。

### 6. Elasticsearch 400 BadRequest 的终极之战

这是整个过程中最棘手的问题。

- **现象**：Elasticsearch 明明在运行，`curl` 可以成功创建索引，但 Python 代码始终报 `BadRequestError(400, 'None')`。
- **排查过程**：
  1. 检查 IK 插件：已安装 (`analysis-ik 8.19.10`)
  2. 用 `curl` 手工测试索引创建：成功
  3. 打印详细异常：`Invalid media-type value on headers [Accept, Content-Type]. Accept version must be either version 8 or 7, but found 9.`
  4. 发现问题！`elasticsearch-py` 库版本是 9.x，它在请求头中声明了 `compatible-with=9`，而 ES 8.x 无法处理，返回 400。
- **解决**：**不再使用 `elasticsearch-py` 进行索引创建和 bulk 写入**，改用 `httpx` 直接向 ES 的 REST API 发送请求。在 `ValueESRepository` 中，`ensure_index` 和 `index` 方法全部用 `httpx.AsyncClient` 实现，完全复制成功的 `curl` 命令。最终 ES 全文索引成功写入 24 条数据。

### 7. 测试脚本期望字段名错误

- **现象**：检索 “销售额” 返回的第一条就是 `order_amount`，但测试却显示 ❌ 未命中。
- **原因**：测试用例中 `expected` 写的是 `"amount"`，而实际字段名是 `"order_amount"`。
- **解决**：将 `expected` 改为配置文件中的真实列名。
